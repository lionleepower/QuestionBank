## 2. 为什么要这样做？

- 解决的实际问题：D2 的 router 只认阿拉伯数字和两个对比关键词，真实中文提问（"一百万规模""mpi和hybrid哪个快"）会漏路由；P1 收尾需要一个**覆盖率有据可查**的 baseline，而不是"感觉能用"。
- 评测集是本项目的核心卖点落地：**"全程可评测"从口号变成一份数据文件**——10 问、每问期望 intent + unknowns，规则 router 今天在上面 10/10，P2 的 LLM router 用同一份文件打分，两个数字放在一张表里就是简历上"对比两种 router 准确率"的证据。
- 服务 Agent-style tool calling：期望值定义在 tool call 层而非答案文本层，意味着评测天然与"router 是规则还是 LLM"解耦——这就是 function calling 里 expected tool + expected arguments 的评测形状。

## 3. 为什么这样设计目录或模块？

- 评测问题放 `evals/questions.jsonl`（数据）而不是写死在测试里（代码）：测试文件是"怎么打分"，jsonl 是"考什么题"——加题不改代码；P2 的 LLM router 评测脚本读同一份文件；jsonl 逐行独立，git diff 按题可读。
- 中文数字逻辑全部留在 `router.py` 内部（`_` 前缀私有函数 + 模块级常量表），不新建 `numerals.py`：当前只有一个消费者，抽模块是过早抽象；表驱动（三个 dict）让"支持哪些字符"一眼可见。
- 文案打磨不动测试：渲染层测试只锁要素（token 在、顺序对），措辞自由——这是 task_007/008 定下的"契约用常量钉死、文案留自由度"边界的直接受益：打磨三处文案，零测试改动。

## 4. 为什么选择这种实现方式？

- 双模式点名是**实体共现**信号，与关键词信号（"对比"）性质不同：用户点名两个可比实体本身就是对比意图的表达。实现为独立分支而非塞进 `COMPARE_KEYWORDS`——"两个都出现"无法表达为单个子串命中（第 7.2/7.3 节两个坑都是想把它硬塞进去导致的）。
- 中文数字解析选**手写累加器**（30 行、纯函数、零依赖）而不是 NLP 库：支持范围画线在"数字 + 十/百/千/万/亿的规范组合"，覆盖评测集与真实提问的头部；已知长尾（"以防万一"会解析出 1）**故意不堵**，作为 Q3"规则边际成本"论据保留。
- 规则全部确定性、可枚举：每条路由都能指出命中哪条规则，错了能写回归测试钉死——这是和 LLM router 对比时 baseline 的"可解释"卖点。

## 5. 为什么不使用其他方案？

- 替代方案 1：现在就上 LLM 做意图识别。
  为什么暂时不用：没有 baseline 的 LLM 评测无从谈"提升了多少"；demo 环境要求可离线、零 key、确定性；P2 上 LLM 时这套规则就是对照组——"先确定性基线、再上 LLM、全程可评测"是项目主线。
- 替代方案 2：jieba 等分词/NLP 库解析中文数字。
  为什么暂时不用：新增依赖要 ask first；需求只是量级词组合，30 行累加器可测试、可讲解（面试能白板推导），库反而把逻辑藏进黑盒。
- 替代方案 3：把双模式规则写成一条大正则（`mpi.*hybrid|hybrid.*mpi`）。
  为什么暂时不用：两个子串测试 + `and` 语义一目了然；正则要处理顺序对称、大小写，可读性差且易错——正则适合"模式"，不适合"共现"。
- 替代方案 4：评测期望值用最终中文答案字符串。
  为什么暂时不用：答案文本耦合渲染措辞，打磨文案就要全量改期望；tool call 层的期望对两种 router、任何文案版本都稳定（Q1 的核心）。

## 7. 结对过程中暴露的追问（本次真实踩过的坑）

以下每条都是写 route 双模式规则时真实犯过的错（三版才写对）。答案由 Claude 代答（2026-07-17）——**模拟面试抽问时要能脱稿复述**，尤其 7.2 的短路求值和 7.3 的字符串迭代语义。

### 7.1 "更快"关键词捷径（第一版）

把 `"更快"` 加进 `COMPARE_KEYWORDS`，三条 mode_mention 测试碰巧全绿——但任务包指定的"双模式点名"规则并没有实现。追问：为什么测试全绿还是错的？"更快"和"对比"在语义类型上差在哪？

答：测试是**欠约束**的——正例恰好都带"哪个更快"，反例恰好不带，于是一个错误实现从缝隙里穿过去了；评测集第 7 问同样没拦住。这是 Q4"只收录能答对的问题的评测集没价值"的活例子：测试/评测通过是必要条件，不是充分条件，**实现对不对最终要对着规格（任务包）验收，而不是对着测试**。语义类型上，"对比/比较"是**对比意图**的直接表达；"更快"是**比较级修饰词**，出现在大量非对比问题里——"有没有比 64 核更快的配置"是求最优（best_config 语义），按第一版会被路由去 compare_modes，回一段答非所问的模式对比。错答比拒答伤害大，这就是它必须撤掉的理由。

### 7.2 `("比较", "对比", "mpi" and "hybrid")`（第二版）

想表达"mpi 和 hybrid 都要出现"，写成了元组元素 `"mpi" and "hybrid"`。追问：这个表达式求值成什么、为什么？`and`/`or` 返回的是布尔还是操作数？`or` 的这个特性在什么惯用法里反而有用？

答：求值成 `"hybrid"`——Python 的 `and`/`or` 是**短路求值且返回操作数本身**，不强转布尔：`x and y` 在 x 为真时返回 y（x 假时返回 x）；`"mpi"` 非空为真，所以整个表达式就是第二个操作数 `"hybrid"`。于是元组实际是 `("比较", "对比", "hybrid")`，"只提 hybrid"的问题被误路由。`or` 的对偶特性（x 真返回 x，否则返回 y）是常见惯用法：`name = user_input or "default"` 用来取默认值——同一个语言特性，放错位置是 bug，放对位置是惯用法。教训：`and` 连接的应该是**两个布尔测试的结果**（`"mpi" in q and "hybrid" in q`），不是两个字符串本身。

### 7.3 `for kw in question` + `kw == "hybrid"`（第三版）

用 for 循环遍历问题、逐个和 "hybrid"/"mpi" 比较计数。追问：`for kw in question` 拿到的是什么？`==` 和 `in` 在这里各是什么语义、各错在哪？英文句子想按词判断，正确做法是什么？

答：字符串迭代产生**单个字符**——`kw` 依次是 "h"、"y"…，Python 字符串没有"按词遍历"。`kw == "hybrid"` 是整体相等测试：单字符永远不等于 6 字符串，恒 False，计数器永远是 0（两个红灯的直接原因）。改成 `kw in "hybrid"` 更危险：`in` 对字符串是**子串/成员测试**，方向反了——它问"这个字符是不是 h/y/b/r/i/d 之一"，"mpi" 里的 `i` 也命中，变成悄悄的假阳性。正确姿势根本不需要循环：`"hybrid" in question.lower()` 一个子串测试完事（`in` 区分大小写，所以先 `lower()` 规范化——case_insensitive 测试考的就是这步）。英文按词判断则先 `split()` 或用正则词边界 `\bword\b`，再做成员测试。

### 7.4 openmpi 会误命中 `"mpi" in q`——可接受吗？

子串测试意味着问题里出现 "openmpi" 一词也算提到 mpi。用户判断：**可接受，长尾留给 P2 的 LLM router**（2026-07-17，用户自己的判断 ✅）。追问：把这个判断展开成完整论证。

答（骨架已由用户给出，展开待口头）：规则 router 负责高频头部——误判**可枚举、可写成回归测试**；"openmpi vs mpi"这类需要词边界甚至语义理解的长尾，每堵一个都要加特判、维护成本递增（这就是 Q3 的边际成本曲线），而 LLM 天然处理它。关键是**同一份评测集**能把"换 LLM 到底赚了多少"量化出来：P2 时往评测集里加入 openmpi、"更快"歧义这类规则已知会错的问题，两种 router 的分差就是 LLM 的增量价值。已知局限 + 明确的升级路径 + 可量化的对比，比"规则写到完美"更像工程判断。

### 7.5 中文数字解析（Claude 代填，返工清单三连）
