## 1. 这个 task 做了什么？

- 主要文件：`hpc_agent/report.py`、`hpc_agent/cli.py`、`tests/test_report.py`、`tests/test_cli.py`

## 2. 为什么要这样做？

- 解决的实际问题：此前每个工具只能回答一个点状问题（谁最快 / 两模式谁强 / 各核数效率如何），**没有一个"把 ARCHER2 扫描结果讲成一个完整故事"的产物**。报告是面试 demo 里第一个可以直接展示的 artifact。
- 服务可复现 data flow：报告字节级确定 → 可以把生成的 `.md` 提交进 `reports/` 当基准，任何改动导致输出漂移都会在 git diff 里现形。
- 服务 Agent-style tool calling：`scaling_report` 是第一个**组合层**——它消费其他 tools 的结构化输出而不重查数据库，这正是将来 LLM orchestration 的形状（LLM 调多个 tool、汇总成自然语言）；现在用确定性代码先把"汇总"这一层做成可评测的 baseline。
- 工作流副产物：本 task 暴露"只有端到端测试时，填好单个空也无法验证"的问题，确立了"每空配单元测试"的两层测试规则。

## 3. 为什么这样设计目录或模块？

- `report.py` 只做**组合与渲染**：不写 SQL（数据来自三个既有工具）、不算指标（speedup/efficiency 由 metrics 层算好）——每层只消费下一层的公开输出。
- 契约用模块常量钉死（`SECTION_*` / `TABLE_HEADER` / `MODES_MISSING_NOTE`），测试 import 同一批常量做断言：改标题只改一处，且"哪些是契约、哪些是自由文案"在文件头一眼可见。
- CLI 只管 IO 边界：解析参数、写文件、退出码；`scaling_report` 本身不接触文件系统，返回 dict——保持纯函数便于测试与将来被 FastAPI / LLM 复用。
- 刻意不放进来的：报告的持久化路径策略（调用方决定写哪）；时间戳/环境信息（破坏确定性，见 Q4）；LLM 生成的解读（W2 之后作为对照叠加，见 Q3）。

## 4. 为什么选择这种实现方式？

- **parts 列表 + 一次 `"\n".join`**：段落顺序即列表顺序，结构一眼可读；固定段落用 `+= [...]` 按块声明，不定长的 bullets 用 `*` 解包摊平，循环逐行攒（表格）用 `append`（语法笔记：`references/notes/python-list-building.md`）。
- **失败语义分两层**：工具层（compare_modes）缺模式整体 `found: False`——对比工具的合同是"给出可比的两边"；报告层同一情况只把该节降级为"数据不足"——报告的合同是"汇总现有信息"，一节缺数据不应扣掉整份报告（Q1 的核心）。
- **解读规则化生成**：阈值、归因、推荐全部由确定性规则产出，同输入必同输出——先有可评测的 baseline，才谈得上 W2 用 LLM 对比提升（Q3）。
- **舍入只在渲染层**：rows 里的全精度值一路传到 f-string 才 `:.2f`；time_sec 是原始测量值，连渲染层都不动（Q5）。

## 5. 为什么不使用其他方案？

- 替代方案 1：模板引擎（Jinja2）渲染报告。
  为什么暂时不用：新增依赖要 ask first；六个 section 的量级上，模板带来的分离收益小于引入的间接层；f-string + 列表拼接零依赖且逐行可断点。报告若长到多页/多格式再迁移。
- 替代方案 2：解读部分交给 LLM 生成。
  为什么暂时不用：不可评测、不确定（同数据不同措辞），会把整份报告的字节级确定性一起破坏；项目的刻意路径是"先确定性基线、再上 LLM、全程可评测"。规则版 bullets 将来就是 LLM 版的对照组。
- 替代方案 3：`scaling_report` 自己写一条大 SQL 一次取齐。
  为什么暂时不用：打破"SQL 只在 db.py"的分层，并复制三个工具已实现的逻辑（tie 规则、模式谓词、基线校验）；组合既有工具让报告自动继承它们的修复和测试（代价收益的完整讨论见 Q2）。
- 替代方案 4：输出 HTML/PDF。
  为什么暂时不用：Markdown 在 GitHub 上直接渲染、git diff 友好、面试 demo 零工具链；格式转换属于展示层扩展，不值得现在付复杂度。

### Q2 组合三个 tools vs 自己写 SQL

`scaling_report` 没有一行 SQL，全部数据来自 `scaling_summary` / `best_config` / `compare_modes`。这样做的收益是什么（提示：分层、测试继承、将来 LLM orchestration 的形状）？代价是什么（提示：三次独立查询、上游 dict 形状变了谁先知道）？什么情况下你会反过来选"一条大 SQL"？

答：确保功能分层，同时方便进行测试，以及功能的继承， 然后不知道（LLM orchestration 的形状）；代价则是三次独立查询的成本比较高，上游的dict, 如scaling_summary变了则会影响到其它的全部组件的正常使用，需要重新的测试与调控；当实现一个全新的功能时，我会选择一条大的SQL

批改：收益半对 ✅ 但术语要打磨："功能的继承"应说**测试与修复的继承**——上游修一个 tie 规则 bug，报告零改动自动受益。"orchestration 形状"补课：scaling_report 手写版干的事（调多个 tool → 拿结构化结果 → 汇总成自然语言）就是 LLM agent loop 里每一轮的形状；W2 把"决策者"换成 LLM 时，三个 tool 一行不改——这就是现在用组合而不是大 SQL 的最大收益。代价部分方向对但重点偏了：本地 SQLite 三次点查的**性能成本可忽略**（task_007 Q2 同款结论，别再把性能当理由），真正的代价是**形状耦合**——上游改 dict 字段，报告在运行时才 KeyError；"谁先知道"的答案是**组合层的端到端测试**（这正是两层测试里 e2e 层存在的意义）。另外方向说反了：scaling_summary 变了影响的是它的**下游消费者**，不是"其它全部组件"。"实现全新功能时选大 SQL"❌ 太泛，站得住的场景是：性能真实成为瓶颈（网络往返、需要库内聚合的大数据量）、需要单条查询的原子一致快照、或该查询是一次性分析而非会被复用的工具。→ 返工后两问。


### Q3 阈值与规则化解读

你最终把效率阈值定为 0.4——但真实数据里 128 核的 efficiency 是 0.4965，这条 bullet 在真报告里**一次都不会触发**；定 0.5 它就会在 128 核处出现。你的选择是哪个、凭什么？另外：解读为什么用规则生成而不是（现在就）交给 LLM——"可评测的 baseline"在 W2 的评测集里具体怎么用到这些规则？

答：现在的数据量其实并不完全足够，另外，0.4其实也是efficiency的正常范围，并不能因为要实现这个功能而去违反现实，而且我们通过在测试当中就已经可以去测试这个功能的实现是否正常，所以我选择0.4；这样做可以帮助模型进行决策的参考，属于确定性路由，LLM则是概率采样，可能出现漂移，同时我们的这个功能测试零依赖，每次路由都能指出命中哪条规则

批改：前半有一个**站得住的种子** ✅："不能因为想让 bullet 触发而去调阈值"——这是反 Goodhart（为了指标好看而扭曲标准）的正确姿势，面试里说得出口。但两处要修：①"0.4 是 efficiency 的正常范围"说不通——efficiency 0.4 意味着六成核在空转，多数 HPC 语境下这不叫正常；工程锚点记两个：0.5（一半浪费，加核性价比分界）、0.7（许多 HPC 中心的 allocation 建议线）。②没答代价：定 0.4 意味着真报告对"128 核效率已腰斩到 0.4965"只字不提——你要能主动补上"这段信息由扩展性表格和超线性 bullet 承担，阈值 bullet 只负责报警更严重的浪费"才算闭环。后半 ❌ 答非所问：你背的是规则 vs LLM 对比表，但问的是**评测集里具体怎么用**——答案：规则版解读是 **golden reference（参考答案）**。W2 让 LLM 生成解读后，按"事实点覆盖"打分：推荐配置对不对、跌破位置对不对、归因方向对不对——这些事实点有唯一可验证的正确值，正是确定性规则提供的。没有规则版，LLM 输出的对错就只能靠人眼判断，评测不成立。→ 返工后半。


### Q4 字节级确定性

docstring 要求同一 db + unknowns 生成字节级一致的 markdown。为什么标准要严到"字节级"而不是"内容大致相同"（提示：测试怎么断言、git diff 怎么用、将来 LLM 版报告拿什么当对照）？代码里其实**没有任何一行**是专门为确定性写的——那确定性是从哪来的？哪些常见写法一旦加进来就会破坏它？

答：这些内容都是关键词查询，所有的规则都是固定，输出也是可控的，生成的模板可供llm进行大致的参考；其它的不知道

批改：❌ 整题返工。"关键词查询"完全不沾边——report 链路（scaling_report → 三个 tools → SQL）里没有任何关键词路由，关键词是 router.py 的事。三个子问的标准答案：①**为什么字节级**：测试断言就是一行 `first == second` 字符串相等，零成本、零歧义；"内容大致相同"没有可执行的定义，断言写不出来。生成的 .md 提交进 `reports/` 后，任何行为漂移都在 git diff 里现形；W2 的 LLM 版报告也需要一个逐字稳定的对照组。②**确定性从哪来**：不是"加"出来的，是**没被破坏**——数据来自带显式 ORDER BY 的 SQL，中间全是纯函数，渲染是固定 f-string，链路上不存在不确定输入。这就是"没有一行代码专门为确定性写"的含义：确定性是默认状态，破坏它才需要动作。③**什么会破坏**：`datetime.now()`（时间戳）、`os.environ` / 主机名、`random`、set 的迭代顺序、无 ORDER BY 的 SQL（SQLite 不保证行序）、依赖文件系统列目录顺序。→ 返工整题。

## 7. 结对过程中暴露的追问（本次 session 实际踩过的坑）

以下每条都是你写代码时真实犯过的错。答案由 Claude 代答（2026-07-14）——**模拟面试抽问时你要能脱稿复述**，尤其 7.1 的错误类型三分法和 7.2 的机制推导。

### 7.1 字面量 vs 名字（同一堵墙撞了三次）

`config[found]`（想要字面量写成变量）→ `"markdown": <str>`（文档记号当代码）→ `"## SECTION_BEST"`（想要变量写成字面量）。追问：三个错分别在什么阶段暴露？为什么暴露时机不同？

答：三个错的暴露阶段依次是**运行时 / 解析时 / 测试时**。`config[found]`：语法合法，运行到这行才发现名字不存在 → `NameError`。`<str>`：`<` `>` 被当成比较运算符但表达式残缺 → 解析阶段 `SyntaxError`，一行都不会执行。`"## SECTION_BEST"`：语法、运行全部正常，产出的只是**错误的文本**——唯一能抓住它的是测试对内容的断言（`md.index(SECTION_BEST)` → ValueError）。教训分两层：判断口诀是"引号里的东西 Python 一个字都不动，写之前问自己要的是字符本身还是名字指向的值"；工程含义是**越晚暴露的错误越依赖测试兜底**——第三类错误没有测试就直接进产物了，这也是"契约钉成常量 + 测试 import 同一常量"的价值：手打字符串的机会被消灭了。

### 7.2 超线性归因写反（"cache miss" vs "cache miss 减少"）

第一版 bullet 写"efficiency > 1 主要可能是 cache miss 或内存负载不均"。追问：为什么这是反的？把正确机制推导一遍。

答：efficiency > 1 = 加 N 倍核快了超过 N 倍 = 每个核比单核时**干活更快**了，所以归因必须是一个"让单位工作变快"的机制。核数增加时聚合 cache 容量同步增加（每核自带 L1/L2）：单核跑全量数据时工作集塞不进 cache、访存频繁；分到多核后每核只持有数据的一个分片，工作集开始**装进 cache，miss 减少**，单位工作变快——这才是超线性的来源。"cache miss（多）"是让程序变慢的东西，解释不了变快；"负载不均"只会拖慢。面试官听到"超线性因为 cache miss"必追问"miss 多了怎么会更快"，当场露馅。附带边界：超线性通常出现在中等核数（工作集恰好跨过 cache 容量线），核数继续加大后通信开销接管，efficiency 回落——样例数据里 2 核 1.47 → 128 核 0.50 正是这个形状。

### 7.3 条件写两遍必分叉

`any(row["total_cores"] > 1.0 ...)` 守门，列表推导却按 `efficiency > 1.0` 过滤——无超线性数据时守门放行、列表为空，输出病句"在配置下"。追问：结构性修法是什么？

答：同一个业务条件出现在两处，改一处忘一处就是时间问题（这次是初版抄错键）。修法是**让条件只存在一处**：先建过滤后的列表，再用列表自身的真值当守门——`superlinear = [...]` + `if superlinear:`。空列表为假，守门与内容天然一致，还省一次遍历。单元测试 `test_bullets_stay_silent_about_superlinear_without_it` 就是为这个坑立的：无超线性数据时不许出现 cache 字样。

### 7.4 阈值照搬与量纲

效率阈值第一版抄了咖啡店示例的 0.05——那是**周环比增长率**的 ±5%，和并行效率（0~1 的比值）量纲完全不同，导致真实数据下规则永远沉默。追问：怎么系统性避免"示例数字被当成默认值"？

答：任何从示例/模板搬来的常量，落地时必须回答"这个数在**我的量纲**里意味着什么"。并行效率的工程常识锚点：0.5（一半核在空转，加核的钱花得值不值的分界）和 0.7（很多 HPC 中心的 allocation 建议线）。检验手段是拿真实数据走一遍规则看它触发在哪——0.05 永不触发、0.4 在本数据集也永不触发（最低 0.4965）、0.5 恰好在 128 核触发。阈值本身没有唯一正确答案，但"选它的理由 + 它在真实数据上的行为"必须能说出口（Q3 正是这题）。

### 7.5 time_sec 被 `:.2f` 破坏原值契约

初版表格六列全 `:.2f`：245.7 → "245.70"、3.866 → "3.87"，两条端到端断言直接挂。追问：`f"{x}"`、`f"{x:.2f}"`、`round(x, 2)` 三者的输出差异？

答：`f"{x}"` = `str(x)`，float 用最短无损表示（245.7 就是 "245.7"）；`f"{x:.2f}"` 四舍五入到两位**并补零**（1.0 → "1.00"，这是 `| 1.00 |` 断言要的）；`round(x, 2)` 返回 float 而非字符串，`str(round(1.0, 2))` 是 "1.0" 不是 "1.00"——想控制显示位数必须走格式规约。契约层面：time_sec 是原始测量值，"3.87" 丢掉的 9 毫秒对 128 核的运行是真实信息（下游可能拿它再算比值）；speedup/efficiency 是派生比值，展示两位足够（完整讨论见 Q5）。

### 7.6 append 列表 = 嵌套一层（`*` 解包）

`parts` 里塞进整个 bullets 列表，`"\n".join` 报 `TypeError: expected str, list found`。追问：append / extend / `+=` / `[*a]` 的选型逻辑？

答：加"一个东西"用 append，并"一串东西"用 extend / `+=` / `*` 解包；`["## 解读", *bullets, ""]` 等价于 append + extend + append 三行。两个伴生坑：extend/`+=` 遇到字符串会拆成单字符（字符串也可迭代）；`+=` 就地修改而 `= lst + [...]` 新建对象，有别名时结果不同。完整笔记：`references/notes/python-list-building.md`。

### 7.7 反斜杠续行把缩进吃进字符串

`"...are\` 换行续写时，下一行的行首缩进**全部成为字符串内容**，报告里出现大段空洞——测试没钉这段措辞所以不报错，产物却是坏的。追问：多行长字符串的干净写法？

答：括号内**隐式拼接**——`("第一段 " "第二段")`，相邻字符串字面量在解析时合并，缩进留在代码里不进内容；或者干脆一行写完。反斜杠续行在字符串内几乎总是错的。这也是"测试只锁要素、不锁全文"的边界演示：宽松断言放过了坏产物，渲染质量要靠 review 兜底（与 task_007 的 7.6 同一结论）。
