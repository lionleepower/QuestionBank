## 1. 这个 task 做了什么？

- `parse_unknowns(text)`：从中文问题里解析规模——"100万" → 1000000（万后缀优先），"unknowns=1000000" 这类 ≥ 4 位裸数字兜底，"128核" 不误匹配，无规模返回 `None`。
- `route(question)`：关键词路由，把问题分类成 intent dict——命中"最快/最优"类关键词 → `{"intent": "best_config", "unknowns": <int|None>}`，其余 → `{"intent": "unknown"}`。
- `ask(question, db_path)`（Claude 代写）：route → tool → 渲染中文回答的完整链路；缺规模时反问，查无数据时转述 tool 的 message。
- `cli.py` / `__main__.py`（Claude 代写）：`import-csv` 和 `ask` 两个子命令，让整条链路能在终端演示。
- 主要文件：`hpc_agent/router.py`（2 个挖空）、`hpc_agent/cli.py`、`hpc_agent/__main__.py`
- 验证命令：`python -m pytest tests/test_router.py tests/test_cli.py`，以及面试现场 demo：

  ```bash
  python -m hpc_agent import-csv examples/size_grid_1000000_sample.csv
  python -m hpc_agent ask "100万规模下最快配置是什么？"
  ```

## 2. 为什么要这样做？

- 解决的实际问题：之前 metrics/tools 只能在 pytest 里被调用，没有面向"人问一句中文"的入口；这个 task 把"CSV 进 → 中文问题进 → 中文答案出"整条链路打通，任何一天被约面都能现场跑。
- 服务可复现 data flow：路由是纯字符串规则、无网络无随机性，同一句话永远走到同一个 tool、同一个 db 永远给同一个答案。
- 服务 Agent-style tool calling：`route()` 输出的 intent dict 就是 tool call 的雏形——intent 对应 tool name，unknowns 对应 tool argument。后续换成 LLM function calling 时，替换的只是"谁来产出这个 dict"，下游 tool 层不动（对应关系见 Q1）。

## 3. 为什么这样设计目录或模块？

- `router.py` 管"理解问题"（解析 + 分类 + 渲染回答），`tools.py` 管"回答问题"（查库 + 结构化结果），`cli.py` 只管参数解析和进程出入口——三层各自可以独立测试。
- 边界：router 不写 SQL、不碰 schema；tools 不懂中文、不关心问题怎么问；cli 不含业务逻辑，薄到几乎不值得测逻辑（所以 CLI 测试测的是 wiring，见 Q4）。
- 刻意不放进来的：LLM 调用（v0 刻意无 LLM，见 Q1）、对比类意图（D3 的 compare_modes 再接入）、报告生成（D4）。

## 4. 为什么选择这种实现方式？

- 两条正则按先后顺序编码优先级："万"后缀语义明确所以先查，裸数字加 `{4,}` 位数下限当兜底（取舍见 Q2）。
- 关键词元组 + `any(kw in question for ...)`：确定性、零依赖、毫秒级，关键词集合是模块级常量，改动只有一处。
- 返回 intent dict 而不是答案字符串：路由的测试不需要数据库，加新 tool 时改动面小（分层意义见 Q3）。
- 错误处理分两种：`ask()` 对"缺规模"反问而不是抛异常，与 task 5 里 `speedup()` 的 fail fast 相反——两种策略各自的适用条件见 Q5。

## 5. 为什么不使用其他方案？

- 替代方案 1：**直接调 LLM API 做意图识别**。暂时不用：引入网络依赖和不确定性，测试没法钉死断言，demo 环境（无网/无 key）直接失效；而且 v0 的教学目标正是先理解"路由 → intent → tool"这层结构本身（深入对比见 Q1）。
- 替代方案 2：**jieba 分词 / 中文 NLP 库做解析**。暂时不用：问题模式高度受限（规模 + 意图关键词），两条正则就能钉死契约；新增重依赖违反 Ask-first 边界，还把可解释的规则变成黑盒。
- 替代方案 3：**click / typer 写 CLI**。暂时不用：标准库 argparse 零依赖够用，两个子命令撑不起一个框架；`main(argv)` 的形式也让测试可以进程内调用（见 Q4）。

## 6. 核心问题 Q1–Q5（week1 验收题）

### Q2 解析规则的优先级与边界

`parse_unknowns` 里为什么"万"优先于裸数字、裸数字为什么要求至少 4 位？举一个会被这两条规则救回来的真实问题。这套规则会在什么输入上出错（至少举一个 false positive）？

答：
先把这类语言转化为可被接受的数字，接下来再去进行实验就会有保障；裸数字超过4位是为了防止客户口胡且利用领域知识排除别的量；"128核"是核数、"4节点"是节点数，而 problem size 至少是万级，位数下限是用领域知识排除别的量；比如说：那个实验最6？这里的6就是一个形容词，不具备其它意思；输入出错也有可能，比如说用户输入"4万万不可", 这里的意图就会被错误识别

批改："4万万不可" 这个 false positive ✅（会解析成 40000，且句子里根本没有规模语义）。但两个主问答偏了：**① 裸数字要 ≥ 4 位**不是防"口胡"，而是这个领域里短数字另有含义——"128核"是核数、"4节点"是节点数，而 problem size 至少是万级，位数下限是用领域知识排除别的量。**② "万"为什么优先**：万后缀语义明确（一定是规模），裸数字只是猜测；被救回来的真实问题就在测试里——"job13996330 里 100万规模谁最快"，若裸数字先查，job id 会被当成规模。→ 返工这两点。


### Q3 intent dict 分层

`route()` 返回 `{"intent": ..., "unknowns": ...}` 而不是直接返回答案字符串——这个分层的意义是什么？D3 要接入 `compare_modes`，在这个设计下需要改哪几行？

答：
这是为了方便之后agent的工具的调用；因为agent不止有一个工具，当用户的目标需要多个工具时，我们就得要function calling,去产出调用请求;parse_unknowns(), route(question: str), 以及ask(question: str, db_path: PathLike)都要改，会多增加一些意图识别的关键词以及分类

批改：方向对但两处不准。**① 分层意义**漏了最实在的一条：route 不碰数据库，所以路由测试零依赖、毫秒级（看 test_route_* 那几个测试——没有 db_path 参数）；intent dict 还让"分类对不对"和"答案对不对"可以分开定位 bug。**② 改哪几行**：`parse_unknowns` 大概率**不用改**（对比句里的规模解析方式相同）；要改的是——tools.py 加 `compare_modes`（D3 你主写）、`route` 加一个关键词分支返回新 intent、`ask` 加一个 dispatch 分支；cli.py **一行不改**。"哪些不用改"和"要改哪些"同样重要，它证明分层是有效的。→ 返工。



### Q4 进程内 CLI 测试 vs subprocess

CLI 测试为什么直接调 `main(argv)` 而不是 `subprocess` 起真进程？两种方式各测到了什么、漏了什么？

答：main(argv)可以直接测试整个工作流程，更小的成本以及更高的速率，会更占优势,subprocess测得是用户真实敲命令的完整体验

批改：**方向反了**。subprocess 起真进程永远比进程内函数调用**更慢**（每个测试都要付解释器启动的钱），和项目大小无关。正确对比——进程内 `main(argv)`：快、可下断点、覆盖率统计得到、容易注入 tmp 路径；**漏测**的是 `python -m` 入口配置（`__main__.py` 写错了测不出来）、真实的 OS 退出码、shell 层面的参数引用/编码。subprocess：测的是用户真实敲命令的完整体验，但慢、难 debug、覆盖率断掉。常见做法：绝大多数用进程内，留一两个 smoke test 走真进程。→ 返工整题。



### Q5 反问 vs raise

`ask()` 缺规模时返回追问文案，而 task 5 的 `speedup()` 遇到坏输入直接 raise——同一个项目里两种相反的错误处理策略，为什么都是对的？（提示：调用方分别是谁？）

答：ask()的调用方是给y用户，缺少完整输入不等于程序非法状态；speedup()是给代码，坏的数据会污染数据库，使得错误值静默流进报告，所以raise;它们都是对的

批改：核心区分（看调用方）抓到了，但两边都说错了对象。**`ask()` 的调用方是人**（终端用户/对话），"没说规模"是正常的对话状态，不是错误——所以反问补槽，对话继续。**`speedup()` 的调用方是程序员的代码**（不是"数据库"），time ≤ 0 说明上游代码或数据有 bug——对 bug 就要 fail fast，立刻崩在离病灶最近的地方，否则错误值静默流进报告。一句话：**用户的不完整输入 ≠ 程序的非法状态**，前者对话解决，后者异常解决。→ 表述返工。

## 7. 结对过程中暴露的追问（本次 session 实际踩过的坑）

### Q6 group(0) vs group(1)

你的第二条正则写了捕获组 `(\d{4,})` 却用 `group(0)` 取值。`group(0)` 和 `group(1)` 分别是什么？这里两者为什么恰好相等？什么时候会不相等（举一个例子）？"写了括号却不用"为什么会被 code review 挑出来？

答：group(0)是整个匹配，group(1) 是第一个捕获组，()的范围不一样时就会被调出来，比如说(\d*)kg，这时group(0)就是25kg, 而group(1)就是25；可读性问题，会误导读者的意图

批改：概念和 kg 例子 ✅。最后一问错了：性能影响可以忽略，**review 挑它不是因为效率，是因为可读性**——捕获组是写给读者的信号："这一段我后面要用"。写了括号却用 group(0)，读者会停下来找"group(1) 去哪了"，找不到就开始怀疑是不是漏了逻辑。代码里每个多余的信号都是读者的负担。→ 修这一句。

### Q7 job id 假阳性

`parse_unknowns("job13996330 谁最快")`——没有"万"、只有 job id——你的实现会返回什么？为什么 `test_parse_unknowns_prefers_wan_over_bare_digits` 那个带 job id 的用例反而能过？如果要防这个 false positive，正则上怎么改（提示：lookbehind / 边界）？

答：13996330;因为那个里面有”万“正好撞上了；(?<![A-Za-z\d])\d{4,}

批改：前两问 ✅（返回 13996330；带 job id 的用例能过是因为"万"分支先命中、先 return）。**`\B` 方案实测双重失败**：

```
\B\d{4,}:
  'unknowns=1000000' -> 匹配 '000000'   # 丢首位，解析成 0！
  'job13996330'      -> 匹配 '13996330' # 照样没防住
```

原因：`\B` 只约束**匹配起始位置**"不是词边界"，`=` 和 `1` 之间是边界所以跳过首位，但引擎会**往后重试**，从第二个数字起 `\B` 就满足了。正确姿势是负向后顾：`(?<![A-Za-z\d])\d{4,}`——要求前一个字符既不是字母也不是数字（禁数字是为了挡住引擎从 job id 中间重试）。教训：**改完正则必须拿正反例都跑一遍**，`re.search` 三行就能验证。→ 返工正则部分。
